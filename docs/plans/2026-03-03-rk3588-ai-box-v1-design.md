# RK3588 AI 算法盒子 V1 设计文档

> 日期：`2026-03-03`  
> 范围：`V1（可上线）`  
> 架构形态：`B/S`  
> 目标硬件：`RK3588 + 16GB RAM + 256GB eMMC`

---

## 1. 已确认业务约束

1. 并发规模：`20 路 IPC` 实时在线分析。
2. 实时预览：`主码流预览`，不使用子码流。
3. 预览链路：`流转发/重封装`，不做解码重编码。
4. AI 分析：主码流解码后进入 `自动抽帧调度`，按并发和负载动态调整。
5. 告警短视频：默认 `20 秒`，前端可配置调整。
6. 存储策略：`滚动覆盖`，媒体空间高低水位 `85% -> 70%`。
7. 跨摄轨迹时间窗：默认 `30 秒`，前端可调。
8. 驾驶舱：需要 `地图/拓扑视图`。
9. 协议能力：支持 `HTTP` 推送、`GAT1400 双向`、`GB28181 接入与级联`。
10. 输入类型：支持 `RTSP/ONVIF`、`MP4 文件分析`、`图片流分析`。
11. 权限要求：`RBAC + 数据范围`，并支持 `tenant/project` 隔离。
12. 账号能力：同一账号支持跨多个项目切换。
13. 登录体系：首版 `本地账号密码`。
14. 品牌能力：`Logo` 和 `系统标题` 可配置。
15. 编码兼容：要求支持 `H.265`，接受“终端能力白名单”策略（不做服务端转码）。

1. 接入层

- 自动抽帧调度
- 媒体切片与落盘
- 协议适配与上报（GAT1400 北向、GB28181 级联）

3. 服务层
- `ingest-hub-service`
- `preview-relay-service`
- `analysis-pipeline-service`
- `event-engine-service`
- `media-recorder-service`
- `protocol-gat1400-service`
- `protocol-gb28181-service`
- `auth-rbac-service`

4. 安全与治理层

- 审计日志（登录、配置变更、规则修改、协议重投）

## 3. 关键数据流

## 3.1 实时预览流

1. IPC 主码流进入 `ingest-hub-service`。
2. 预览分支进入 `preview-relay-service`。
3. 执行封装转换（transmux/remux），不重编码。
4. Web 端根据终端能力选择播放通道（H.264/H.265）。

## 3.2 实时分析流

1. 同一主码流进入分析分支。
2. MPP 解码 + RGA 预处理。
3. 调度器分配当前路分析 FPS（动态）。
4. RKNN 推理 + 跟踪 -> 事件引擎。
5. 事件触发后输出告警记录、关键帧和短视频。

## 3.3 协议互通流

1. 内部统一事件模型输出到 `protocol-gat1400-service` 北向上报。
2. 外部 GAT1400 南向事件接入后映射为内部模型。
3. GB28181 接入与级联通过 `protocol-gb28181-service` 统一管理。
4. 所有协议事件都统一走审计与重试队列。

## 4. 自动抽帧调度设计

## 4.1 调度目标

1. 保证 20 路在线时预览稳定优先。
2. 在资源可承受范围内最大化分析效果。
3. 高风险/告警通道优先。

## 4.2 调度输入指标

1. `npu_utilization`
2. `pipeline_queue_depth`
3. `inference_p95_ms`
4. `soc_temperature`
5. `active_camera_count`
6. `camera_priority`（重点位/普通位）
7. `alarm_density`

## 4.3 调度规则（默认）

1. 基线：每路 `8 FPS`。
2. 中压：降至 `5 FPS`。
3. 高压：普通路降至 `2~3 FPS`，重点路保 `6~8 FPS`。
4. 告警路临时升帧：`10~12 FPS`，持续 `10~20 秒`。
5. 温控触发：全局比例降帧。

---

## 5. H.265 兼容策略（无转码）

1. 服务端不做重编码，仅做转发/重封装。
2. 前端首次登录执行终端能力探测（浏览器/OS/GPU）。
3. 维护项目级“终端能力白名单”。
4. 不支持 H.265 的终端给出明确提示与替代建议。
5. 替代路径为设备侧双码流或终端白名单限制，不引入盒子转码负担。

---

## 6. 存储与清理策略

1. 元数据：`PostgreSQL`。
2. 媒体：本地文件系统（后续可接 MinIO）。
3. 目录建议：
- `data/media/{tenant}/{project}/{camera}/{yyyyMMdd}/{event_id}/`
4. 清理策略：
- 当媒体分区使用率达到 `85%`，启动“最旧优先”清理。
- 释放至 `70%` 后停止。
5. 元数据保留应独立于媒体，防止审计台账缺失。

---

## 7. 权限与隔离模型

## 7.1 多租户多项目

1. 层级：`tenant -> project -> site -> camera`。
2. 核心表强制带 `tenant_id`、`project_id`。
3. 所有查询默认附带 tenant/project 过滤。

## 7.2 RBAC + 数据范围

1. 角色示例：
- `super_admin`
- `tenant_admin`
- `project_admin`
- `operator`
- `analyst`
- `auditor`
- `api_client`
2. 权限维度：
- 页面权限
- API 权限
- 操作权限
- 数据范围权限（site/camera）

## 7.3 跨项目切换

1. 用户与项目通过 membership 关联。
2. 每次请求必须携带当前项目上下文并校验授权。
3. 切换项目需重签发实时通道令牌并写审计日志。

## 9. 非功能指标（V1 建议）

1. 告警链路端到端时延：`<= 2s`（单路基线）。
2. 20 路持续稳定运行：`7x24`。
3. 预览可用率：`>= 99.5%`。
4. 告警落盘成功率：`>= 99%`。
5. 协议上报成功率（含重试）：`>= 99%`。
