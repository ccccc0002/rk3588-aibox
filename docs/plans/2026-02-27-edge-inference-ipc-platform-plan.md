## 1. 项目目标与约束

### 1.1 业务目标
1. 接入 IPC 网络摄像头（RTSP/ONVIF），实现稳定拉流和断线恢复。
2. 实时检测人/车并触发告警，落盘告警事件、关键帧、告警短视频。
3. 提供检索能力：
4. 以文字搜历史告警视频。
5. 以文字搜图片。
6. 以图片搜图片。
7. 构建历史轨迹（同摄连续轨迹 + 跨摄拼接轨迹）。

### 1.2 硬件约束（固定）
1. 边缘平台芯片：`RK3588`
2. 内存：`16GB`
3. 本地存储：`eMMC 256GB`
4. 推理模式：`全边缘推理`（在线链路不依赖云端推理）

### 1.3 技术参考库（指定）
1. 视频编解码（MPP）：`https://github.com/HermanChen/mpp`
2. 色域/缩放转换（RGA）：`https://github.com/airockchip/librga`
3. AI 推理（RKNN）：`https://github.com/airockchip/rknn_model_zoo`

## 2. 总体架构（方案 B：全边缘推理）

### 2.1 服务划分
1. `camera-access-service`：IPC 管理、ONVIF 探测、RTSP 拉流、健康检查。
2. `video-pipeline-service`：MPP 解码、RGA 色域转换/缩放、帧缓存。
3. `inference-service`：RKNN 模型推理（人/车检测）与目标跟踪。
4. `alert-engine-service`：规则判断、事件生成、告警去抖与合并。
5. `media-recorder-service`：事件关键帧抓拍、告警短视频切片。
6. `feature-index-service`：图像特征向量、文本向量生成与索引检索。
7. `trajectory-service`：轨迹拼接、轨迹检索、轨迹可视化数据输出。
8. `api-gateway`：统一查询 API（检索、告警、轨迹、设备管理）。
9. `edge-console`：本地运维与业务页面。

### 2.2 关键数据流
1. RTSP 拉流 -> MPP 解码 -> RGA 转换 -> RKNN 推理 -> 告警判定。
2. 触发告警 -> 保存关键帧与短视频 -> 写入事件数据库。
3. 从关键帧提取 embedding -> 写入向量索引。
4. 用户检索（文/图）-> 向量召回 -> 映射事件与视频 -> 返回结果。
5. 轨迹服务按时间窗、摄像头拓扑、特征相似度完成拼接。

## 3. RK3588 技术落地方案

### 3.1 视频解码（MPP）
1. 采用 `mpp` 对 RTSP 流进行硬件解码，目标格式优先 `NV12`。
2. 统一解码输出分辨率到推理所需尺寸（或交由 RGA 缩放）。
3. 建立“解码线程 + 帧队列 + 背压策略”，防止堆积导致 OOM。
4. 每路摄像头维护独立状态机：连接中、拉流中、重连中、异常中。

### 3.2 色域转换与缩放（RGA）
1. 使用 `librga` 完成 NV12 -> RGB/BGR 转换。
2. 使用 RGA 进行 resize、crop、letterbox，尽量避免 CPU memcpy。
3. 保持“零拷贝/少拷贝”路径，统一帧内存管理（池化复用）。

### 3.3 AI 推理（RKNN）
1. 以 `rknn_model_zoo` 为基线，优先落地人车检测模型。
2. 模型输入尺寸建议先定为 `640x640`，在性能测试后再调优。
3. 推理后处理包括 NMS、类别筛选、置信阈值过滤。
4. 跟踪采用 ByteTrack（CPU 轻量实现），维持稳定 `track_id`。
5. 后续如引入多模态模型，优先使用可在 RK3588/NPU 上稳定运行版本。

### 3.4 性能预算建议（首版）
1. 单路建议推理帧率：`8~12 FPS`（业务感知与资源平衡）。
2. 摄像头接入目标建议分级：
3. S 级（10~20 路）：首版上线验证。
4. M 级（30~50 路）：优化后目标。
5. L 级（>50 路）：需进一步模型量化和调度优化。
6. eMMC 容量控制：默认保存告警片段，不保存全量原始流。

## 4. 核心功能设计

## 4.1 告警事件闭环
1. 规则维度：区域入侵、目标出现、停留超时、频次超阈。
2. 事件结构：event_id、camera_id、target_type、score、time_range、rule_id。
3. 告警视频：前后缓冲合成（例如前 5 秒 + 后 5 秒）。
4. 关键帧：事件首帧、峰值帧、结束帧。

## 4.2 多模态检索
1. 文搜图：文本 embedding -> 图像 embedding ANN 检索。
2. 图搜图：图片 embedding -> 图像 embedding ANN 检索。
3. 文搜视频：文搜图召回事件 -> 事件映射短视频。
4. 结果重排：相似度 + 时间优先 + 摄像头范围过滤。

## 4.3 历史轨迹
1. 同摄轨迹：按 track_id 聚合连续片段。
2. 跨摄轨迹：按时间窗 + 拓扑邻接 + 特征相似度拼接。
3. 输出形式：时间轴、摄像头跳转链、关键帧预览、关联视频。

## 5. 数据与存储设计

### 5.1 元数据存储
1. 建议数据库：`PostgreSQL`
2. 关键表：
3. `camera`：设备信息、拉流配置、在线状态。
4. `event`：告警事件主表。
5. `event_media`：关键帧和短视频索引。
6. `detection`：帧级检测结果。
7. `track`：轨迹片段表。
8. `trajectory_link`：跨摄拼接关系表。

### 5.2 向量存储
1. 小规模先用 `pgvector`（简化部署）。
2. 后期规模扩大可切换 `Milvus`。
3. embedding 表需保留模型版本字段，便于重建索引。

### 5.3 媒体存储
1. 本地对象存储建议 MinIO（部署在边缘机）。
2. 目录规范：
3. `D:\edge-inference-ipc-platform\data\media\{site}\{camera}\{yyyyMMdd}\{event_id}\`
4. 生命周期：
5. 热数据（检索高频）：30 天。
6. 冷数据（归档可查）：90~180 天（按容量策略调整）。

## 10. 风险与对策

1. 夜间/逆光导致误检漏检。
2. 对策：场景化阈值 + 样本回流 + 模型微调。
3. eMMC 写入压力与寿命风险。
4. 对策：顺序写、分层缓存、日志轮转、生命周期清理。
5. 多路并发造成内存抖动。
6. 对策：固定帧队列长度 + 内存池 + 动态降载。
7. 跨摄轨迹误拼接。
8. 对策：引入摄像头拓扑约束与时间窗门控。
