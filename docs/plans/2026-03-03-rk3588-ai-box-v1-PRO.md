# RK3588 AI Box V1 PRO

> 文档类型：`PRO (Project Readiness & Operations)`  
> 版本：`v1.0`  
> 日期：`2026-03-03`  
> 适用范围：`研发 / 测试 / 运维 / 项目交付`

---

## 1. 文档目标

本 PRO 用于统一 V1 项目的研发与测试执行口径，确保以下目标可落地：

1. 研发可按统一边界开发，不出现模块职责重叠或遗漏。
2. 测试可按统一验收标准执行，不出现“开发通过、测试无标准”。
3. 联调可按统一接口契约推进，不出现协议字段口径冲突。
4. 上线可按统一发布与回滚流程执行，不出现临场决策风险。

---

## 2. V1 范围与边界

## 2.1 已确认范围（In Scope）

1. `20 路 IPC` 在线接入与分析。
2. 主码流实时预览，预览链路仅转发/重封装，不做服务端转码。
3. 自动抽帧分析调度（按负载与优先级动态调整）。
4. 输入源支持：RTSP/ONVIF、GB28181、MP4 文件、图片流。
5. 协议支持：HTTP 推送、WebSocket、GAT1400 双向、GB28181 接入与级联。
6. 驾驶舱支持地图/拓扑视图与跨摄轨迹展示。
7. 权限体系：本地账号密码 + RBAC + tenant/project 隔离 + 跨项目切换。
8. 品牌配置：Logo、系统标题可配置。

## 2.2 非范围（Out of Scope）

1. 云端推理依赖（V1 为全边缘推理）。
2. 盒子内视频转码服务（明确不做 decode->encode 转码）。
3. 高阶算法能力（如大模型多模态复杂推理）不纳入 V1 必达。

---

## 3. 系统基线参数（冻结）

1. 并发规模：`20 路`。
2. 告警短视频：默认 `20 秒`，前端可调（建议限制 6~30 秒）。
3. 存储策略：滚动覆盖，媒体水位 `85% -> 70%`。
4. 跨摄轨迹时间窗：默认 `30 秒`（建议可调 10~120 秒）。
5. H.265：支持，采用终端能力白名单策略（不转码兜底）。
6. 架构形态：B/S，分层页面，不做单页堆叠。

---

## 4. 架构与模块职责

## 4.1 核心服务

1. `ingest-hub-service`
- 职责：视频接入、单路拉流多路扇出、流健康管理。

2. `preview-relay-service`
- 职责：预览转发、remux/transmux、终端播放能力协商。
- 约束：禁止编码器路径。

3. `analysis-pipeline-service`
- 职责：MPP 解码、RGA 预处理、RKNN 推理、跟踪。

4. `adaptive-scheduler`
- 职责：基于负载指标动态分配每路分析 FPS。

5. `event-engine-service`
- 职责：规则计算、事件去抖、事件合并、统一事件模型输出。

6. `media-recorder-service`
- 职责：关键帧抓拍、告警视频切片、媒体索引。

7. `protocol-gat1400-service`
- 职责：GAT1400 南向接入与北向上报、ACK/重试/死信。

8. `protocol-gb28181-service`
- 职责：GB28181 接入、目录同步、通道管理、国标级联。

9. `auth-rbac-service`
- 职责：本地账号认证、角色权限、项目上下文校验、审计。

10. `console-api-service`
- 职责：统一管理 API、查询 API、配置 API。

11. `cockpit-ui`
- 职责：分中心页面展示、实时告警、驾驶舱、系统管理。

## 4.2 分层页面信息架构

1. 实时中心：预览墙、在线状态、抽帧状态。
2. 分析中心：实时告警、历史事件、轨迹分析、规则管理。
3. 任务中心：MP4 分析、图片流分析、任务结果。
4. 协议中心：GAT1400/GB28181/HTTP/WS 配置与报文追踪。
5. 驾驶舱：KPI、地图/拓扑、趋势图、跨摄链路。
6. 系统管理：用户角色、项目隔离、品牌配置、审计日志。

## 5. 接口与数据契约（对齐基线）

## 5.1 统一事件模型（内部）

```json
{
  "event_id": "evt_20260303_xxx",
  "source_type": "rtsp|gb28181|mp4|image|gat1400",
  "tenant_id": "t001",
  "project_id": "p001",
  "camera_id": "cam_001",
  "event_type": "intrusion|stay|line_cross|custom",
  "target_type": "person|vehicle|other",
  "track_id": "trk_12345",
  "score": 0.92,
  "bbox": [120, 60, 360, 300],
  "start_ts": "2026-03-03T10:00:00Z",
  "end_ts": "2026-03-03T10:00:20Z",
  "snapshot_url": "/media/.../snap.jpg",
  "clip_url": "/media/.../clip.mp4",
  "version": 1
}
```

## 5.2 事件推送要求

1. 幂等键：`event_id + target_endpoint`。
2. 语义：至少一次投递（at-least-once）。
3. 失败策略：指数退避 + 最大重试 + 死信队列。
4. 可观测：每次推送要有 trace_id 与状态日志。

## 5.3 权限上下文要求

1. 每次 API 请求必须带 `tenant_id/project_id` 上下文。
2. 后端二次校验用户 membership。
3. 不允许跨项目无权限查询。

## 6. 研发执行规范

## 6.1 DoR（Definition of Ready）

研发任务开始前必须满足：

1. 需求边界已明确（有 In/Out Scope）。
2. 接口契约已冻结（字段、错误码、鉴权方式）。
3. 测试用例已至少覆盖主路径与关键异常路径。
4. 依赖关系清晰（上游/下游/配置项/数据准备）。

## 6.2 DoD（Definition of Done）

研发任务完成必须满足：

1. 单元测试通过。
2. 集成测试通过。
3. 关键日志、指标与告警项已接入。
4. 文档更新完成（接口、配置、部署、回滚）。
5. 测试可复现执行路径。

## 6.3 分支与提交流程

1. 分支命名：`feature/*`、`fix/*`、`release/*`。
2. 提交规范：`type(scope): summary`。
3. 合并要求：至少 1 次代码评审 + 测试记录。
4. 严禁直接改动 `main` 未评审上线。

## 7. 测试策略（全链路）

## 7.1 测试层级

1. 单元测试：模型调度、权限中间件、协议映射。
2. 集成测试：接入->分析->事件->落盘->推送闭环。
3. 协议联调：GAT1400 双向、GB28181 接入与级联。
4. 性能测试：20 路并发、P95 时延、资源占用。
5. 稳定性测试：7x24 连续运行、异常恢复。
6. 安全测试：鉴权绕过、越权访问、审计可追溯。

## 7.2 核心测试矩阵

1. 视频链路
- H.264/H.265 主码流预览可用性
- 无转码约束验证（禁止 encoder 调用）

2. 抽帧调度
- 正常负载：8 FPS 目标
- 中压/高压：降帧生效
- 告警场景：优先升帧

3. 协议能力
- GAT1400 南向接收映射正确
- GAT1400 北向上报 ACK/重试正确
- GB28181 注册/保活/目录/级联正确

4. 权限隔离
- 跨项目访问拒绝
- 项目切换重签名与审计记录

5. 存储策略
- 到达 85% 自动清理
- 清理至 70% 停止
- 元数据完整性不受损

## 7.3 验收指标（V1）

1. 告警端到端时延：`<= 2s`（单路基线）。
2. 预览可用率：`>= 99.5%`。
3. 告警落盘成功率：`>= 99%`。
4. 协议上报成功率（含重试）：`>= 99%`。
5. 20 路连续稳定运行通过。

## 9. 发布与回滚机制

## 9.1 发布前检查

1. 所有阻塞级缺陷关闭。
2. 关键 KPI 达标。
3. 协议联调签字完成。
4. 数据备份与回滚预案验证。

## 9.2 回滚策略

1. 服务版本回滚：保留 N-1 版本包与配置快照。
2. 配置回滚：支持按版本恢复（协议、调度、权限）。
3. 模型回滚：保留上一个稳定模型与校验摘要。

---

## 10. 风险清单与应对

1. H.265 终端兼容波动
- 对策：能力白名单 + 终端指引 + 项目级可见性告警。

2. 高并发下预览影响分析
- 对策：单拉多用、调度降帧、队列背压与优先级策略。

3. 协议重试风暴
- 对策：幂等键、重试上限、死信队列、人工重放。

4. 跨项目数据泄漏
- 对策：中间件强制项目上下文 + 数据层双重校验 + 审计。

5. eMMC 寿命与写放大
- 对策：滚动覆盖、顺序写、日志轮转、冷数据外置归档。
